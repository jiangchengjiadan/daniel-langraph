# Trip.Planner 可复用实现拆解

日期：2026-04-27

本文目标不是复刻 Trip.com Trip.Planner，而是把它公开可见的产品机制，拆成适合教学项目落地的最小实现。

## 1. 先说结论

Trip.Planner 公开信息说明，它不是“一个大模型直接生成一段旅行文案”，而是四层组合：

1. 轻输入层  
   目的地、天数、风格等少量输入。
2. 结构化规划层  
   输出的是 itinerary，不是纯文本。
3. 约束与数据层  
   结合开放时间、交通估算、价格、可用性、POI 数据。
4. 编辑与交易层  
   用户可以改、替换、删除、重排，并关联预订。

对于教学项目，建议只复用前 3 层中的“轻量版本”，不做完整交易闭环。

## 2. Trip.Planner 可推断的实现结构

基于 Trip.com 官方页与新闻稿，可以较稳地推断：

- LLM 负责：
  - 根据出行风格生成行程草案
  - 解释推荐原因
  - 处理用户的自然语言调整请求

- 规则/结构化层负责：
  - 按天切分行程
  - 控制每天景点数量
  - 约束开放时间和典型游览时长
  - 控制城市内/城市间交通合理性

- 平台数据层负责：
  - POI、酒店、餐厅、交通库存
  - 价格、评分、可用性
  - 已有订单导入

- 前端负责：
  - itinerary 画布
  - 地图联动
  - 增量编辑

这里最重要的一点是：**大模型不是主控一切，而是被结构化数据和规则约束。**

## 3. 教学项目可复用的最小实现

如果要在 `daniel-trip-agent` 里参考 Trip.Planner，我建议只做以下最小版本。

### 3.1 输入层

前端改成更轻的规划入口：

- 多个城市：可选择或手动输入
- 出发日期
- 旅行天数
- 旅行风格

折叠高级项：
- 交通方式
- 住宿偏好
- 额外要求

这样既接近 Trip.Planner，也不会让前端表单过重。

### 3.2 请求模型

后端从单城市升级到多城市：

```json
{
  "cities": ["上海", "苏州", "杭州"],
  "start_date": "2026-05-01",
  "travel_days": 4,
  "style": "文化 + 城市漫步"
}
```

第一期不要做复杂最优路线，只按用户输入顺序规划。

### 3.3 规划层

保留现有 LangGraph，不切到 Deep Agents。

最小改造方式：

1. 将总旅行天数按城市顺序分配
2. 每个城市复用当前单城市规划能力
3. 汇总为一个总 itinerary
4. 在城市切换处补一条“城际交通段”

这层可以继续复用：
- 景点搜索
- 酒店搜索
- 天气查询
- FlyAI 酒店/门票增强
- 商品回填

### 3.4 约束层

教学版只做最基本约束：

- 每天 2-3 个景点
- 酒店类型与住宿偏好大致匹配
- 下雨天少安排强户外景点
- FlyAI 价格异常时不覆盖原预算
- 多城市按输入顺序串联

不要做：
- 复杂路线优化
- TSP/多目标优化
- 实时库存闭环
- 精细到分钟级的时窗调度

### 3.5 前端结果页

参考 Trip.Planner 的不是视觉细节，而是交互结构：

- 左侧：天级 itinerary
- 右侧：地图或城市摘要
- 每一天展示：
  - 城市
  - 酒店
  - 景点
  - 天气
  - 预算小计

第一期不做拖拽，只做：
- 删除景点
- 替换酒店
- 重新生成某一天

## 4. 为什么不建议直接上 Deep Agents

这个项目当前核心是“确定性旅行规划工作流”，不是“开放式长任务协作”。

现有代码中，很多关键能力都属于稳定业务逻辑：

- 高德 POI / 酒店 / 天气查询
- FlyAI 商品增强
- 商品回填
- 预算重算
- 价格校验

这些更适合继续放在 LangGraph 节点和普通函数里，而不是交给 Deep Agent 自由决定。

Deep Agents 更适合第二阶段再做：
- 多轮需求澄清
- 多方案比较
- 长期偏好记忆
- 输出多个版本的旅行方案

第一阶段没有必要引入这层复杂度。

## 5. 建议的工程拆分

### 后端

- `TripRequest` 支持 `cities: List[str]`
- `TripPlan` 支持 day 对应城市字段
- LangGraph state 增加 `cities`、`current_city`
- 增加多城市分配逻辑
- 保留现有节点不大改

### 前端

- 首页输入区重做为 Trip.Planner 风格
- 增加多城市输入组件
- 结果页改为 itinerary 工作台式布局

## 6. 第一阶段验收标准

如果按教学项目标准，我建议第一阶段只验这几条：

1. 用户可输入多个城市
2. 系统按输入顺序输出多城市 itinerary
3. 每天能看到城市、景点、酒店、天气、预算
4. 页面风格从“聊天工具”更像“旅行规划器”
5. 不引入 Deep Agents，也不引入复杂优化器

## 7. 最终判断

Trip.Planner 最值得借的，不是“AI 很强”，而是：

- 极简输入
- 结构化输出
- 受约束生成
- 可编辑规划界面

对你们这个教学项目，最合理的实现方式是：

**保留现有 LangGraph 后端，前端参考 Trip.Planner 的输入与结果布局，同时把单城市升级成按顺序串联的多城市轻量规划器。**
