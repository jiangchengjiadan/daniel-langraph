顺序流是图的最简单形式。数据从节点 A 流到节点 B。在 LangGraph 中，当节点 A 完成时，它返回的数据会**合并**到主状态中。然后，这个**更新后**的主状态会被传递给节点 B。

## 2.1 状态定义：使用 TypedDict 定制化共享状态

自定义状态是构建复杂 Agent 的基础。通过 TypedDict 定义状态，我们为整个图提供了一个明确的类型契约，所有节点都必须遵守。这种方式不仅提升了代码的可读性，也确保了数据结构的一致性。

## 2.2 节点实现：函数式更新状态与增量更新

节点函数接收完整的状态对象，执行其计算逻辑，然后返回一个包含所需更新的字典。理解节点函数的返回值是**增量更新（Delta Update）**而非整个新状态至关重要。

LangGraph 在底层负责将这些更新安全地合并到中央状态中。这意味着即使 Python 函数的引用可能在函数内部被修改（例如，操作传入的字典），但从 LangGraph 的角度来看，它只关注返回的更新字典。

这种机制是实现状态一致性和后续 Reducer 机制的基础。

## 2.3 固定边缘：设置 START、END 和串联流程 (add_edge)

线性流程的定义依赖于固定的边缘。流程的起点可以使用 `set_entry_point("node_name")` 来指定第一个执行的节点，或者更灵活地使用 `add_edge(START, "node_name")`。

虽然 `set_entry_point` 对于简单流程足够，但使用 `add_edge(START,...)` 更具扩展性，因为它可以配合条件路由函数来创建**条件入口点**，允许流程根据初始状态动态选择第一个执行的节点。

终点则通过 `add_edge(..., END)` 显式标记，通知 LangGraph 流程结束。

## 实战代码

我们将构建一个"诗歌创作链"：

**节点A (generate_topic):** 生成一个诗歌主题
**节点B (write_poem):** 根据主题写一首诗

```python
from typing import TypedDict
from langgraph.graph import StateGraph, END
from langchain_openai import ChatOpenAI
from langchain_core.messages import HumanMessage

# --- 0. 设置 ---
# 确保你设置了 OPENAI_API_KEY
llm = ChatOpenAI(temperature=0)

# --- 1. 定义状态 (State) ---
class PoemState(TypedDict):
    topic: str     # 诗歌主题
    poem: str      # 诗歌内容

# --- 2. 定义节点 (Nodes) ---
def generate_topic(state: PoemState):
    print(f"--- 正在执行 generate_topic ---")
    prompt = "给我一个关于夏天的、适合写诗的随机主题，只返回主题词。"
    response = llm.invoke([HumanMessage(content=prompt)])
    print(f"生成的主题: {response.content}")
    # 返回一个字典来更新状态
    return {"topic": response.content}

def write_poem(state: PoemState):
    print(f"--- 正在执行 write_poem ---")
    # 从 state 中读取上一步生成的 topic
    topic = state['topic']
    prompt = f"请根据主题\"{topic}\"写一首五行短诗。"
    response = llm.invoke([HumanMessage(content=prompt)])
    print(f"生成的诗歌:\n{response.content}")
    # 返回一个字典来更新状态
    return {"poem": response.content}

# --- 3. 定义图 (Graph) ---
workflow = StateGraph(PoemState)

# 添加节点
workflow.add_node("generate_topic", generate_topic)
workflow.add_node("write_poem", write_poem)

# --- 4. 定义图的结构 (Edges) ---
# 设置入口点
workflow.set_entry_point("generate_topic")

# 添加边：
# 1. generate_topic 之后 -> write_poem
workflow.add_edge("generate_topic", "write_poem")
# 2. write_poem 之后 -> END
workflow.add_edge("write_poem", END)

# --- 5. 编译图 ---
app = workflow.compile()

# --- 6. 运行图 ---
# 初始状态可以为空，因为第一个节点会生成数据
initial_state = {}
final_state = app.invoke(initial_state)

print("\n--- 最终结果 ---")
print(final_state)

# --- 预期输出 (示例) ---
# --- 正在执行 generate_topic ---
# 生成的主题: 蝉鸣的午后
# --- 正在执行 write_poem ---
# 生成的诗歌:
# 烈日高悬，
# 蝉鸣声声噪，
# 绿荫下微风，
# 冰镇西瓜甜，
# 悠然一夏。
#
# --- 最终结果 ---
# {'topic': '蝉鸣的午后', 'poem': '烈日高悬，\n蝉鸣声声噪，\n绿荫下微风，\n冰镇西瓜甜，\n悠然一夏。'}
```

## 本节重点

**状态合并 (State Merging):**
节点A (generate_topic) 返回 `{"topic": "..."}`，这个字典会**合并**到 PoemState 中。

**数据传递:**
节点B (write_poem) 之所以能工作，是因为它从 state 参数中读取到了 `state['topic']`，这是上一步的输出。

**add_edge("A", "B"):**
建立了从 A 到 B 的固定流向。
